# Technical Risk Checklist

这是原“阶段 C”里的技术审查内容，现改为按需引用的参考清单。

## 何时引用

- 阶段 2 比较方案时，需要补充技术风险判断。
- 阶段 3 输出断点和改动时，需要补充风险备注。

## 审查维度

| 维度 | 关注点 | 典型风险 |
|---|---|---|
| 数据结构 | 怎么存、字段语义是否清楚 | 大 key、结构混乱 |
| 查询模式 | 读写频率、是否有索引 | 全表扫、超时 |
| 并发安全 | 多请求同时修改会怎样 | 覆盖写、竞态 |
| 数据增长 | 时间长了会不会爆 | 只写不删 |
| 缓存策略 | 是否需要缓存、过期是否合理 | 脏数据、永不过期 |
| 错误处理 | 下游失败会怎样 | 直接 500、无兜底 |
| 接口设计 | 参数、返回值、分页 | 返回全量、接口脆弱 |
| 兼容性 | 老客户端 / 老数据是否能工作 | 新老版本冲突 |

## 使用原则

- 这里只做“风险提醒”，不替代主阶段结构。
- 如果风险不影响当前主路径，不要把它抬成主结论。
- 如果风险会直接影响推荐方案落地，必须在阶段 2 或阶段 3 里明确指出。
