# Solution Selection Reference

阶段 2 的核心不是“多写几个方案”，而是把方案筛出来。

## 固定筛选顺序

1. **先看能不能达成最终目标**
   - 方案做完后，用户能不能真的得到想要的结果？
   - 不能，就直接淘汰。

2. **再看是不是主路径优先级**
   - 这个方案是在解决主路径问题，还是只在优化边角？
   - 不是主路径，就降级，不要放在 P0。

3. **最后看简单性**
   - 在都能达标的前提下，默认优先选最简单的方案。

## 简单性的判断标准

- 改动文件更少
- 新增概念更少
- 对现有链路侵入更小
- 后续维护和解释成本更低

## 常见比较维度

| 维度 | 问法 | 默认倾向 |
|---|---|---|
| 本地解决还是调下游 | 能自己算就别麻烦别人？ | 能本地就不调远程 |
| 扩展还是新写 | 现有能力改改能不能用？ | 能扩展就别新起一套 |
| 复用还是新增字段 | 语义是不是同一件事？ | 语义一致再复用 |
| 同步还是异步 | 用户是不是必须立刻看到？ | 必须立刻看到才同步 |
| 配置化还是硬编码 | 以后会不会频繁新增同类需求？ | 高频再配置化 |

## 输出要求

- 方案数控制在 `1~3` 个。
- 每个方案都要给出“推荐 / 降级 / 淘汰”的结论。
- 推荐理由必须回答：
  - 为什么能达成最终目标？
  - 为什么符合当前主路径优先级？
  - 为什么已经足够简单？
