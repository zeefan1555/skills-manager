# Code Review Guidelines Reference

定义这个 skill 在读代码、写 spec 文件夹内各阶段文档时的证据规则与表达方式。

## 证据规则

- 尽量引用具体 `文件:行号`。
- 不确定的判断要写“待确认”，不要装作确定。
- 断点、限制、改动点都要有代码证据或明确的链路依据。

## 断点定义

“断点”指的是：在当前链路里，会阻止推荐方案落地的具体位置。

一个合格的断点说明至少回答：

- 现在这里在做什么？
- 为什么它会挡住推荐方案？
- 需要怎样的改动才能继续？

## 改动项写法

改动清单建议至少包含：

| 文件 | 改动点 | 原因 | 影响范围 |
|---|---|---|---|

## 表达风格

- 先说人话，再给术语。
- 先讲用户会遇到什么，再讲代码为什么会这样。
- 判断要克制，不要把猜测写成结论。

## 不该做的事

- 不凭空发明代码行为。
- 不在没有证据时写“必须这样改”。
- 不把技术风险清单当成主阶段正文重复一遍。
