# Goal-to-Spec：从目标到技术规格

## 这个 skill 做什么

用户给出最终目标，分三个阶段渐进产出 `spec.md`：

1. **阶段 A：看懂项目** — 理解代码库，用大白话梳理全貌，画出用户体验流程图
2. **阶段 B：跑通链路** — 基于 A 的流程图，到代码库里逐环节定位，找出要改的代码和链路断点
3. **阶段 C：审查实现** — 基于 B 的改动清单，做技术选型和排雷

**不执行任何代码变更**，只往 spec.md 写分析文档。

## 工作模式

```
用户给出目标
      |
用户："阶段 A"  →  AI 侦察代码库 → 写 spec.md 的阶段 A 章节
      |
用户审阅 & 反馈（可修正后重跑阶段 A）
      |
用户："阶段 B"  →  AI 读 spec.md 阶段 A 的流程图 → 找代码 → 写阶段 B 章节
      |
用户审阅 & 反馈
      |
用户："阶段 C"  →  AI 读 spec.md 阶段 B 的改动清单 → 技术审查 → 写阶段 C 章节
      |
spec.md 完成
```

### 关键规则

- **用户触发**：用户说"阶段 A/B/C"时才执行对应阶段，不自动推进
- **渐进写入**：每个阶段把内容直接写入 spec.md 的对应章节，不等最后汇总
- **阶段间依赖**：B 必须读 A 已写入的内容，C 必须读 B 已写入的内容
- **可重跑**：用户对某阶段输出不满意，可要求重跑，AI 覆盖该章节
- **按需 brainstorming**：任何阶段内如有细节不明确，调用 brainstorming skill 与用户对齐

## 核心思想：先通后优

为什么分两层（B 和 C）：
- **先通后优**。链路不通，技术实现再漂亮也没用——用户看不到
- **避免遗漏**。混在一起分析时，AI 容易只关注"技术上怎么写"，忽略"写完了用户能不能真的用到"
- **用户能参与判断**。链路层用户凭产品直觉就能判断对不对，技术层才需要深度

| 问题类型 | 案例 | 属于哪一层 |
|----------|------|-----------|
| 新增 widget 但调度器不认识新类型，用户永远看不到 | 链路断点 | 阶段 B |
| 功能跑通了但数据全塞一个 key，查询越来越慢 | 技术隐患 | 阶段 C |
| 新增接口但客户端没有调用入口 | 链路断点 | 阶段 B |
| 接口能调通但没做分页，数据量大时超时 | 技术隐患 | 阶段 C |

## 费曼学习法

整个过程中，解释方式遵循费曼学习法：
- **先说人话，再给术语**。"每个请求进来前先过'安检门'（middleware）"
- **用类比**。goroutine = 同时派出去干活的小分队，channel = 小分队之间传纸条的管道
- **从用户视角讲流程**。不是"Router 分发到 Handler"，而是"用户点了按钮，前台看你要办什么事，把你领到对应柜台"

## 首次触发：确认目标 + 初始化 spec.md

用户首次提出目标时（还没说"阶段 A"），先做两件事：

### 1. 确认目标

- **最终效果**：做完之后用户能看到什么变化？
- **边界**：做什么、不做什么
- **约束**：技术栈、时间、兼容性
- **用户的想法**：已有的思路，采纳并扩展

目标清晰就不追问，最多 2-3 个关键问题。如有细节不清，调用 brainstorming skill 对齐。

### 2. 创建 spec.md 骨架

在项目根目录（或用户指定路径）创建 `spec.md`，写入头部和目标章节：

```markdown
# 技术规格：{目标一句话}

> 生成时间：{日期}
> 状态：进行中

## 目标

### 最终效果
{做完后用户能看到什么}

### 范围
- **做**：...
- **不做**：...

### 约束
- ...

### 用户的想法
{原样保留}

---

<!-- 以下章节由各阶段渐进填充 -->
```

然后告诉用户："spec.md 已创建。准备好了说'阶段 A'，我来侦察代码库。"

## 阶段 A：看懂项目

> 用户说"阶段 A"时执行。完成后将内容写入 spec.md 的「阶段 A」章节。

### 做什么

侦察代码库，用大白话梳理项目全貌，产出用户体验流程图。

### A1. 侦察代码库

#### 工具使用

| 侦察目标 | 用什么工具 | 怎么用 |
|----------|-----------|--------|
| 目录结构 | Glob `**/*` + Bash `ls` | 了解项目布局 |
| 入口文件 | Glob `**/main.*`, `**/index.*`, `**/app.*` | 找到程序入口 |
| 路由/API | Grep 搜索 `router`, `route`, `endpoint`, `handler` | 找到请求入口 |
| 数据模型 | Grep 搜索 `schema`, `model`, `struct`, `interface` | 理解数据结构 |
| 配置文件 | Glob `**/*.{yml,yaml,toml,json,env}` | 了解配置和环境 |
| 调度机制 | Grep 搜索 `cron`, `schedule`, `timer`, `listener`, `subscribe` | 找触发/调度逻辑 |
| 深度探索 | Agent (Explore) | 规模大或调用链复杂时使用 |

#### 按规模调整深度

| 规模 | 判断标准 | 策略 |
|------|----------|------|
| 小型（<50 文件） | 脚本/工具 | 快速通读 |
| 中型（50-500 文件） | 典型服务 | 聚焦目标相关模块 |
| 大型（>500 文件） | 复杂系统 | 只看直接相关部分，用 Explore agent |

#### 侦察重点

| 侦察项 | 做什么 | 为什么 |
|--------|--------|--------|
| 目录结构 | 了解布局 | 知道东西在哪 |
| 入口追踪 | 从入口顺调用链追到底，**重点理清服务间的接口调用关系** | 画服务架构时序图的基础 |
| 数据流 | 数据怎么流转 | 判断改动波及范围 |
| **触发/调度机制** | 定时任务、事件监听、推送逻辑、缓存刷新 | **链路走查的关键——很多断点出在这里** |
| **数据存储方式** | 用了什么数据库、怎么存、key 设计、索引 | **技术审查的关键——很多性能坑出在这里** |
| 已有模式 | 项目里做类似事情的惯例 | 新代码要跟风格一致 |

#### "够了"的标准

能回答这 4 个问题就够了：
1. 用户目标涉及哪些文件和函数？
2. 一个典型操作从头到尾经过哪些服务、各服务之间通过什么接口交互？
3. 新功能从产生到用户看到，经过哪些环节？
4. 数据存在哪、怎么存的、有没有明显的结构问题？

### A2. 产出内容

侦察完成后，将以下内容写入 spec.md 的「阶段 A」章节：

```markdown
## 阶段 A：项目全貌

### A1. 概括
{3-5 句大白话，像跟不懂技术的朋友介绍}

### A2. 服务架构时序图
{Mermaid sequenceDiagram，**聚焦服务间交互**，不涉及服务内部实现细节}

**画图原则**：
- participant 只出现**服务级别**的实体（如：客户端、前端、各后端服务、外部服务）
- 箭头标注的是**服务间的接口调用**（如接口名），不是内部函数调用
- 用 Note 分隔不同阶段（如"触发""处理""通知""展示"）
- 如果有并行请求，用 `par ... and ... end` 表达
- 不画存储组件（DB/Redis/MQ 等），这些留给 A3 详细流程

### A3. 核心流程（含组件交互）
{Mermaid sequenceDiagram，在 A2 基础上展开，加入存储/MQ/配置等组件的交互}
{讲故事方式，标注每步做什么}

### A4. 用户体验流程图
{从用户操作起点到看到结果的完整链路，每一步标注经过的代码位置}

```
用户体验链路 = 触发 → 传递 → 处理 → 存储 → 通知/刷新 → 展示
```

{Mermaid sequenceDiagram，环节编号与阶段 B 链路段一一对应}

### A5. 概念翻译

| 术语 | 大白话 | 类比 |
|------|--------|------|
| ... | ... | ... |
```

**这个阶段最重要的产出是 A4 用户体验流程图**——阶段 B 将基于这张图逐环节到代码库里定位。

### A 阶段完成后

告诉用户："阶段 A 已写入 spec.md。看看流程图对不对？没问题说'阶段 B'，我顺着流程图去找代码。"

## 阶段 B：跑通链路（能不能用？）

> 用户说"阶段 B"时执行。先读 spec.md 中阶段 A 的内容（特别是 A4 用户体验流程图），然后写入「阶段 B」章节。

### 做什么

**核心问题：新功能做完之后，用户真的能看到/用到吗？**

基于阶段 A 的用户体验流程图，到代码库里逐环节验证：每个环节的代码在哪、能不能支撑新功能、哪里会断。

### B1. 逐环节定位代码

对 A4 流程图中的每个环节：
1. 到代码库里找到对应的文件和函数（用 Grep / Read）
2. 读懂这段代码现在做什么
3. 判断新功能在这一步需要什么改动
4. 标注这一步是"通"还是"断"

### B2. 逐环节验证

对每个环节问 6 个问题：

| 验证问题 | 用大白话说就是 |
|----------|---------------|
| **会被触发吗？** | 现有的"闹钟"（定时任务）/ "开关"（事件）能触发新功能吗？还是需要新加一个？ |
| **数据能流过去吗？** | 中间有没有"过滤器"把新数据挡住了？数据结构认识新类型吗？ |
| **能到达用户吗？** | 推送/轮询/缓存刷新机制能把新内容送到用户眼前吗？ |
| **用户能看到吗？** | 客户端/前端知道怎么显示新内容吗？ |
| **老用户怎么办？** | 没更新的用户遇到新数据会崩溃还是正常忽略？ |
| **出错了呢？** | 某一环挂了，整个链路会怎样？有兜底吗？ |

### B3. 产出内容

写入 spec.md 的「阶段 B」章节：

```markdown
## 阶段 B：链路走查（能不能用？）

### B1. 链路验证图
{基于 A4 的流程图，标注每个环节的代码位置 + 通/断状态}
{Mermaid 时序图，断的环节用红色/虚线标注}

### B2. 链路断点

#### 断点 1：{标题}
- **位置**：{文件:函数}
- **现状**：{现在怎么做的}
- **问题**：{为什么新功能在这断了}
- **用户影响**：{不修的话用户会怎样}
- **修复方向**：{大概怎么修}

#### 断点 2：...

### B3. 要改的代码清单

| 文件 | 改什么 | 为什么要改 | 链路环节 |
|------|--------|-----------|---------|
| ... | ... | ... | 触发/处理/存储/展示 |
```

**这个阶段最重要的产出是 B3 要改的代码清单**——阶段 C 将基于这个清单逐项做技术审查。

### B 阶段完成后

告诉用户："阶段 B 已写入 spec.md，共 N 个断点，需要改 M 处代码。看看有没有遗漏？没问题说'阶段 C'，我对每个改动做技术审查。"

---

## 阶段 C：审查实现（好不好用？）

> 用户说"阶段 C"时执行。先读 spec.md 中阶段 B 的内容（特别是 B3 改动清单），然后写入「阶段 C」章节。

### 做什么

**核心问题：怎么实现最合理？实现方式有没有埋雷？**

对阶段 B 的每个改动点做两件事：
1. **选型**：有哪些实现路径？各自代价？推荐哪个？
2. **排雷**：选定的方案有没有技术隐患？

### C1. 实现方案选型

对 B3 列出的每个改动，不要直接给方案，先列出可选路径做对比。

常见选型维度（按需选用）：

| 选型问题 | 用大白话说就是 | 选择倾向 |
|----------|---------------|----------|
| **调不调下游？** | 能自己算就别麻烦别人 | 能本地解决就不调远程 |
| **新增字段还是复用？** | 已有字段能搭便车吗？ | 语义一致就复用，语义不同就新增 |
| **新写还是扩展？** | 现有代码改改能用吗？ | 逻辑相似就扩展，差异大就新写 |
| **同步还是异步？** | 用户要不要等？ | 要立刻看到 → 同步；可以晚几秒 → 异步 |
| **推还是拉？** | 主动通知还是等用户来取？ | 时效高 → 推；可延迟 → 拉 |
| **客户端还是服务端？** | 逻辑放哪边？ | 展示相关 → 客户端；数据相关 → 服务端 |
| **存哪里？** | DB、缓存、还是内存？ | 要持久 → DB；要快 → 缓存；临时 → Redis+TTL |
| **配置化还是硬编码？** | 以后还会加类似的吗？ | 会 → 配置化；不会 → 硬编码 |

如有选型拿不准，调用 brainstorming skill 与用户讨论。

### C2. 技术排雷

选定方案后，审查有没有技术隐患：

| 审查维度 | 关注什么 | 典型的坑 |
|----------|----------|----------|
| **数据结构** | 怎么存？key 设计合理吗？ | 所有数据塞一个 key → 大 key |
| **查询模式** | 读写频率、有没有索引 | 每次查全表没索引 → 超时 |
| **并发安全** | 多个请求同时操作 | 两人同时改同一条 → 覆盖 |
| **数据量增长** | 随时间会怎样 | 只写不删 → 半年后爆了 |
| **缓存策略** | 该缓存吗？过期策略对吗？ | 永不过期 → 看到旧数据 |
| **错误处理** | 失败了怎么办 | 下游挂了直接 500 |
| **接口设计** | 参数、返回值、分页 | 返回全量 → 数据多了超时 |

### C3. 扫描已有代码债务

新功能可能放大已有问题：
- 新功能增加了数据量 → 已有的慢查询更慢
- 新功能增加了调用频率 → 没限流的接口会被打爆
- 已有的"临时方案"一直没改 → 新功能叠上去后不可维护

### C4. 产出内容

写入 spec.md 的「阶段 C」章节：

```markdown
## 阶段 C：技术审查（好不好用？）

### C1. 实现方案选型

#### 改动点 1：{要改什么}

| 方案 | 做法 | 优点 | 缺点 | 改动量 |
|------|------|------|------|--------|
| A | ... | ... | ... | N 个文件 |
| B | ... | ... | ... | M 个文件 |
| **推荐** | {推荐哪个，为什么} ||||

#### 改动点 2：...

### C2. 技术隐患

#### 隐患 1：{标题}
- **位置**：{文件:函数}
- **风险**：{会导致什么问题}
- **什么时候会爆**：{触发条件}
- **改进方案**：{怎么改}

### C3. 现有代码债务
{已有的问题 + 新功能会不会放大}

### C4. 实施路径

| 优先级 | 做什么 | 解决什么 | 复杂度 |
|--------|--------|----------|--------|
| P0 | 修复断点 1, 2 | 链路跑通，用户能看到新功能 | {评估} |
| P1 | 修复隐患 1 | 避免性能/安全问题 | {评估} |
| P2 | 还债 | 长期可维护 | {评估} |

### C5. 还没想清楚的问题
{需要进一步讨论的}
```

### C 阶段完成后

告诉用户："spec.md 全部完成。三个阶段的分析都已写入。可以基于这份 spec 进入实施了。"

## 注意事项

- **不执行任何代码变更**。只往 spec.md 写分析文档。
- **两层分析不要混**。阶段 B 只管"能不能用"（链路通不通），阶段 C 才管"好不好用"（代码质量）。
- **用大白话**。术语第一次出现时翻译 + 类比。
- **阶段 B 重点关注触发机制**。定时任务、事件监听、推送逻辑、缓存刷新——大部分"加了但用户看不到"的问题出在这些地方。
- **阶段 C 重点关注数据**。怎么存、怎么查、数据量增长后会怎样——大部分"能用但越来越慢"的问题出在数据层。
- **分析要有代码证据**。引用具体 `文件:行号`，不凭空想象。不确定的标"待确认"。
- **方案要可执行**。"优化性能"不是方案，"将 SaveAll() 的单 document 存储拆分为按 widget_id 独立 document + 建索引"才是。
- **每个阶段写完立刻写入 spec.md**。不要在对话中输出完整内容再写文件——直接写文件，对话中只给摘要。
