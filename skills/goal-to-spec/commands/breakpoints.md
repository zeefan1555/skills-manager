# Breakpoints Command

> **引用规范**：`../references/spec-template.md`、`../references/code-review-guidelines.md`、`../references/technical-risk-checklist.md`

## Trigger

- `阶段3`
- `breakpoints`
- “分析断点和代码改动”
- “如果选这个方案，代码要改哪里”
- “给我出改动清单”

## 目标

基于阶段 2 已选定的推荐方案，分析当前链路中哪些地方会挡住它落地，并给出代码改动清单。

## 输入

- `research` 的现状调研
- `options` 的推荐方案
- 相关代码证据

## 执行步骤

1. 先明确这次分析基于哪一个“已选方案”。
2. 对照当前现状，找出该方案落地时的链路断点。
3. 为每个断点定位具体代码位置。
4. 说明为什么这里会挡住方案落地。
5. 汇总需要改动的文件、改动原因、影响范围。
6. 将结果写入总文件夹内单独的阶段文档，例如 `03-breakpoints.md`。
7. 必要时补充少量技术风险备注，但不要喧宾夺主。

## 输出

建议写入 `03-breakpoints.md` 的结构：

- `## 阶段 3：断点与代码改动`
- `### 目标链路与当前链路差异`
- `### 断点列表`
- `### 代码改动清单`
- `### 风险备注 / 待确认项`

建议断点至少包含：

- 位置：`文件:行号`
- 当前行为
- 为什么挡住推荐方案
- 需要的改动
- 影响范围

## 返回给用户

- 最关键的断点数量
- 最重要的改动文件
- 还需要确认的点

## 约束

- 不重新做方案选型。
- 不把整个阶段重新写成“技术审查大纲”。
- 技术风险只作为补充备注，不作为主结构。
