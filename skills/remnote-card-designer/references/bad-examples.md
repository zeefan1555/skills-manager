# 反面案例集 — 低质量卡片模式与改写

本文件展示常见的低质量卡片模式。生成卡片时如果拿不准质量，对照这里检查。

---

## 反面模式 1：空泛的概念卡

**差的卡片**：
```
- [Why-Not] 为什么需要缓存？ >>>
    - [场景] 系统需要高性能
    - [问题] 直接查数据库太慢
    - [方案] 加了缓存
    - [代价] 需要维护成本
    - [判断] 性能很重要
```

**问题**：每一行都是废话。"系统需要高性能"不是场景，"直接查数据库太慢"没有量化，"需要维护成本"是句正确的废话。面试官听到这种回答会直接追问"能具体说说吗？"——然后你就卡住了。

**好的卡片**：
```
- [Why-Not] 为什么不直接查 MySQL 主库，而要引入 Redis 缓存层？ >>>
    - [场景] 商品详情页 QPS 峰值 50k，每次查询涉及 3 张表 JOIN
    - [问题] MySQL 单实例扛不住，P99 延迟从 20ms 飙到 800ms，且无法通过加从库线性扩展（JOIN 查询无法拆分）
    - [方案] 将聚合后的商品快照写入 Redis，命中率 > 95%，P99 回到 5ms
    - [代价] 引入缓存一致性问题——价格更新后最长有 30s 的不一致窗口；缓存穿透时需要额外的布隆过滤器
    - [判断] 商品详情页对一致性容忍度高（用户刷新即可），但对延迟零容忍（直接影响转化率）
    - [原理] 缓存本质是用空间换时间+用一致性换吞吐量，关键是判断业务对一致性的容忍度 #[[Extra Card Detail]]
```

## 反面模式 2：缺少场景铺垫

**差的卡片**：
```
- [Trade-off] 使用 eventual consistency 的代价是什么？ >>>
    - [问题] 数据可能不一致
    - [方案] 最终一致性
    - [代价] 短期数据不一致
```

**问题**：没有 `[场景]`，面试官不知道你在说哪个系统的最终一致性。脱离场景的 trade-off 分析毫无意义——所有分布式系统都有这个问题。

**好的卡片**：
```
- [Trade-off] 跨机房订单同步选择最终一致性，核心代价是什么？ >>>
    - [场景] 电商系统双活部署在两个机房，订单写入后需要双向同步
    - [问题] 强一致性要求跨机房 2PC，单次写入延迟从 5ms 涨到 80ms（跨机房 RTT），且任一机房故障会阻塞另一个
    - [方案] 改为异步复制 + 冲突检测：写入只走本机房，异步同步到对端，冲突时以时间戳较新者为准
    - [代价] 同一订单在两个机房短暂不一致（窗口 ~200ms）；极端情况下双写冲突需要人工介入
    - [判断] 订单查询频率远高于修改频率，200ms 的不一致窗口用户几乎感知不到，但写入延迟从 80ms 降到 5ms 直接影响下单成功率
    - [原理] CAP 定理的实际工程选择——在网络分区（机房间链路）不可避免时，选 AP 还是 CP 取决于业务对一致性的真实需求 #[[Extra Card Detail]]
```

## 反面模式 3：全局故事卡变成摘要

**差的卡片**：
```
- [Story] 这篇文章讲了什么？ >>>
    - [背景] 文章介绍了一个分布式存储系统
    - [思路] 使用了 Raft 协议和 LSM-Tree
    - [取舍] 性能和一致性有 trade-off
```

**问题**：这是摘要，不是故事。面试官问"讲讲你对这个系统的理解"，你不能说"它使用了 Raft 协议和 LSM-Tree"——这只是技术栈列举。

**好的卡片**：
```
- [Story] 用 2-3 句话讲清楚 TiKV 的核心设计思路 >>>
    - [背景] 传统单机数据库扛不住 TB 级数据 + 高并发写入，分库分表方案运维成本爆炸（每次扩容要重新分片 + 迁移数据）
    - [思路] 用 Raft 做多副本强一致（解决可靠性），用 Region 分片 + 自动调度（解决扩容痛点），底层用 LSM-Tree（解决写入吞吐）
    - [取舍] 获得了自动扩缩容和强一致，但付出了：跨 Region 事务延迟较高（需要 2PC）、Raft 日志同步带来的写放大（约 3x）
    - [原理] 核心矛盾是"强一致 vs 扩展性"——Raft 保证单 Region 内强一致，Region 分片解决水平扩展，两者解耦是关键设计点 #[[Extra Card Detail]]
```

## 反面模式 4：追问预备卡没有展示横向视野

**差的卡片**：
```
- [Follow-up] 如果面试官问为什么不用 MongoDB？ >>>
    - [区别] MongoDB 是文档数据库
    - [不选] 不适合这个场景
    - [适用] 适合文档存储
```

**问题**：每一行都缺乏深度。"MongoDB 是文档数据库"是事实但不是区别，"不适合"没说为什么，"适合文档存储"是同义反复。

**好的卡片**：
```
- [Follow-up] 如果面试官追问：为什么不用 MongoDB 的分片集群来解决扩展性？ >>>
    - [区别] MongoDB 分片基于 hash/range key，分片后跨分片查询退化为 scatter-gather；TiKV 的 Region 自动分裂+PD 调度让分片对上层透明
    - [不选] 我们的业务有大量跨分片的范围查询（按时间范围查订单），MongoDB 的 scatter-gather 模式下 P99 会随分片数线性增长
    - [适用] MongoDB 分片更适合按单一 key 查询为主的场景（如用户画像库按 user_id 分片），schema-free 也适合快速迭代的早期业务
    - [原理] 分片策略的本质区别：静态分片（MongoDB）vs 动态分片（TiKV），动态分片运维成本低但系统复杂度高 #[[Extra Card Detail]]
```

## 自检口诀

写完每张卡后，快速过一遍：

1. **场景能定位吗？** —— 面试官听完 `[场景]` 知道你在说哪个系统的什么问题吗？
2. **数字有吗？** —— 至少一个具体数字（QPS、延迟、数据量、比例）？
3. **因果链通吗？** —— `[问题]` → `[方案]` 是直接因果关系，还是跳了步？
4. **删一行试试？** —— 如果删掉某一行不影响核心论点，那这一行就不该存在。
5. **说出来顺吗？** —— 想象自己在面试中把这张卡的答案说出来，通顺吗？
