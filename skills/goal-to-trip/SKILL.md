# Goal-to-Trip：从旅行目标到完整行程规格

## 这个 skill 做什么

用户给出旅行目标（去哪、什么时候、玩几天），分三个阶段渐进产出 `trip-spec.md`：

1. **阶段 A：侦察目的地** — 搜集目的地信息，梳理旅行全貌，画出旅行体验流程图
2. **阶段 B：跑通行程** — 基于 A 的流程图，逐日安排行程，找出链路断点（签证过不了？交通接不上？时间排不开？）
3. **阶段 C：审查排雷** — 基于 B 的行程表，做预算估算、风险排查、优化建议

**不代订任何机票酒店**，只往 trip-spec.md 写规划文档。

## 工作模式

```
用户给出旅行目标
      |
用户："阶段 A"  →  AI 侦察目的地 → 写 trip-spec.md 的阶段 A 章节
      |
用户审阅 & 反馈（可修正后重跑阶段 A）
      |
用户："阶段 B"  →  AI 读阶段 A 的流程图 → 安排逐日行程 → 写阶段 B 章节
      |
用户审阅 & 反馈
      |
用户："阶段 C"  →  AI 读阶段 B 的行程表 → 预算风险审查 → 写阶段 C 章节
      |
trip-spec.md 完成
```

### 关键规则

- **用户触发**：用户说"阶段 A/B/C"时才执行对应阶段，不自动推进
- **渐进写入**：每个阶段把内容直接写入 trip-spec.md 的对应章节，不等最后汇总
- **阶段间依赖**：B 必须读 A 已写入的内容，C 必须读 B 已写入的内容
- **可重跑**：用户对某阶段输出不满意，可要求重跑，AI 覆盖该章节
- **搜索验证**：每个阶段都应使用搜索工具获取最新信息（航班、签证政策、景点开放时间等），不凭记忆编造
- **快速模式**：如果用户说"直接全部跑完"或"一次性出完"，则依次执行 A→B→C 一口气写完

## 核心思想：先通后优

为什么分两层（B 和 C），和技术方案设计一个道理：
- **先通后优**。行程走不通，酒店订得再好也白搭——你人到不了
- **避免遗漏**。混在一起规划时，AI 容易只关注"哪好玩"，忽略"去得了吗？回得来吗？时间够吗？"
- **用户能参与判断**。行程层用户凭生活常识就能判断对不对，预算风险层才需要深入分析

| 问题类型 | 案例 | 属于哪一层 |
|----------|------|-----------|
| 到了才发现签证政策变了，入不了境 | 链路断点 | 阶段 B |
| 行程能跑通但每天暴走 3 万步，第三天人就废了 | 体验隐患 | 阶段 C |
| 两个景点之间没有直达交通，硬排在同一天 | 链路断点 | 阶段 B |
| 行程能走但全程吃路边摊，预算反而花在打车上 | 预算分配不合理 | 阶段 C |
| 酒店到机场没有早班车，赶不上回程航班 | 链路断点 | 阶段 B |
| 雨季出行没带伞，户外行程全泡汤 | 风险预案缺失 | 阶段 C |

## 费曼学习法

整个过程中，解释方式遵循费曼学习法：
- **先说人话，再给术语**。"到了先坐城市轻轨（BTS）去酒店"
- **用类比**。BTS = 曼谷版的深圳地铁高架版，Grab = 泰国版滴滴
- **从旅客视角讲流程**。不是"素万那普机场位于曼谷东部 30 公里处"，而是"下了飞机出航站楼，打个 Grab 半小时就到市区酒店"

## 首次触发：确认目标 + 初始化 trip-spec.md

用户首次提出旅行目标时（还没说"阶段 A"），先做两件事：

### 1. 确认目标

- **目的地**：去哪？只去一个城市还是多城联游？
- **时间**：出发日期、回程日期、总共几天几晚
- **出发地**：从哪出发？（影响机票和交通）
- **人数与同行**：一人独行？情侣？家庭？朋友团？（影响酒店、行程节奏）
- **旅行风格**：文化历史？海滨度假？购物美食？冒险探索？全都要？
- **预算范围**：经济型 / 舒适型 / 品质型 / 不设限
- **用户已有想法**：已经想好的部分，原样采纳

目标清晰就不追问，最多 2-3 个关键问题。优先从对话上下文提取信息，减少追问。

### 2. 创建 trip-spec.md 骨架

```markdown
# 旅行规格：{目的地} {天数}天自由行

> 生成时间：{日期}
> 状态：进行中

## 目标

### 行程概要
- **出发地**：{城市}
- **目的地**：{城市}
- **日期**：{出发日} — {返回日}（共 X 天 Y 晚）
- **人数**：{人数及构成}
- **风格**：{旅行风格}

### 范围
- **做**：...
- **不做**：...

### 约束
- ...

### 用户的想法
{原样保留}

---

<!-- 以下章节由各阶段渐进填充 -->
```

然后告诉用户："trip-spec.md 已创建。准备好了说'阶段 A'，我来侦察目的地。"

## 阶段 A：侦察目的地

> 用户说"阶段 A"时执行。完成后将内容写入 trip-spec.md 的「阶段 A」章节。

### 做什么

搜集目的地信息，用大白话梳理旅行全貌，产出旅行体验流程图。

### A1. 信息侦察

#### 侦察清单

| 侦察目标 | 怎么做 | 为什么 |
|----------|--------|--------|
| 天气与季节 | 搜索目的地当月天气、温度、降雨 | 决定穿什么、带什么、能玩什么 |
| 签证政策 | 搜索最新入境政策（免签？落地签？电子签？） | 去不了一切白搭 |
| 航班信息 | 搜索直飞/转机航线、航空公司、大致价格 | 确定出行可行性和成本 |
| 核心景点 | 搜索必去景点、开放时间、门票价格 | 行程规划的骨架 |
| 交通方式 | 搜索市内交通（地铁/公交/打车）、城际交通 | 决定行程节奏和可达性 |
| 住宿区域 | 搜索推荐住宿区域、各区域特点 | 选对位置省一半时间 |
| 美食特色 | 搜索当地必吃美食、推荐餐厅 | 吃好是旅行幸福感的核心 |
| 文化禁忌 | 搜索当地习俗、着装要求、禁忌 | 避免尴尬或冒犯 |
| 货币与支付 | 搜索汇率、换汇方式、电子支付普及度 | 到了才发现没现金就尴尬了 |
| 通讯 | 搜索 SIM 卡 / eSIM / WiFi 方案 | 没网等于瞎了 |

#### "够了"的标准

能回答这 5 个问题就够了：
1. 这个季节去那里，天气怎么样？需要特别准备什么？
2. 入境需要什么手续？有没有最近的政策变化？
3. 怎么去？飞多久？大概多少钱？
4. 到了之后怎么在城市里移动？去周边城市方便吗？
5. 有哪些必去的地方、必吃的东西、必做的体验？

### A2. 产出内容

写入 trip-spec.md 的「阶段 A」章节：

```markdown
## 阶段 A：目的地全貌

### A1. 概括
{3-5 句大白话，像跟朋友安利这个旅行目的地}

### A2. 旅行架构时序图
{Mermaid sequenceDiagram，聚焦"出发地 → 航班 → 目的地城市 → 周边城市 → 返程"的宏观流程}

**画图原则**：
- participant 只出现大的实体：出发城市、航空公司、目的地城市、周边城市、机场
- 箭头标注的是交通方式和大致时间
- 用 Note 分隔不同阶段（出发、游玩、返程）

### A3. 核心流程（含关键组件交互）
{Mermaid sequenceDiagram，展开 A2，加入酒店、交通工具、景点、餐厅等具体组件}

### A4. 旅行体验流程图
{从"决定出发"到"回到家"的完整链路，每一步标注关键动作}

| 环节编号 | 阶段 | 具体步骤 | 关键动作 |
|---------|------|---------|---------|
| ① | 行前准备 | 订机票 | {航线、价格} |
| ② | 行前准备 | 订酒店 | {区域、价格} |
| ... | ... | ... | ... |

**这是阶段 A 最重要的产出** — 阶段 B 将基于这张表逐日展开。

### A5. 目的地小词典

| 术语 | 大白话 | 类比 |
|------|--------|------|
| ... | ... | 像深圳的 xxx |
```

### A 阶段完成后

告诉用户："阶段 A 已写入 trip-spec.md。看看目的地信息有没有遗漏？没问题说'阶段 B'，我来排逐日行程。"

## 阶段 B：跑通行程（能不能走通？）

> 用户说"阶段 B"时执行。先读 trip-spec.md 中阶段 A 的内容（特别是 A4 旅行体验流程图），然后写入「阶段 B」章节。

### 做什么

**核心问题：这个行程从头到尾能走通吗？每天安排合理吗？**

基于阶段 A 的旅行体验流程图，逐日展开具体安排，验证每个环节衔接是否顺畅。

### B1. 机票方案

搜索并列出往返航班方案，包括：

| 方向 | 日期 | 推荐航线 | 航空公司 | 飞行时间 | 参考价格 |
|------|------|---------|---------|---------|---------|

**给出机票建议**：推荐哪个、为什么、在哪买、注意什么（行李额、廉航坑等）。

### B2. 逐日行程表

对每一天，用统一格式输出：

```markdown
#### 🏷️ Day N | M 月 D 日（周X）— {当日主题}

**主题**：{一句话概括}

| 时间 | 安排 | 备注 |
|------|------|------|
| 8:00 | ... | ... |
| 10:00 | ... | ... |
| ... | ... | ... |
```

#### 逐日验证清单

每天的行程需要通过 6 项检查：

| 验证问题 | 用大白话说就是 |
|----------|---------------|
| **交通接得上吗？** | 上一个地方到下一个地方，有没有车？要多久？会不会堵？ |
| **时间够用吗？** | 每个景点待多久合理？赶不赶？走路多不多？ |
| **开门吗？** | 景点那天开不开？有没有特殊闭馆日？需要预约吗？ |
| **吃得上吗？** | 午餐晚餐在哪解决？景点附近有吃的吗？ |
| **体力扛得住吗？** | 今天总共走多少步？有没有连续暴走？需不需要安排休息？ |
| **下雨/极端天气怎么办？** | 今天的行程有没有室内备选方案？ |

### B3. 链路断点

列出行程中所有"走不通"的地方：

```markdown
### B3. 链路断点

#### 断点 1：{标题}
- **位置**：Day X，{时间段}
- **现状**：{现在的安排}
- **问题**：{为什么走不通}
- **旅客影响**：{不改的话会怎样}
- **修复方案**：{怎么调整}
```

### B4. 住宿方案

基于行程推荐住宿区域和更换策略：

| 日期 | 住宿区域 | 推荐理由 | 参考价格 |
|------|---------|---------|---------|

### B 阶段完成后

告诉用户："阶段 B 已写入 trip-spec.md，共 N 天行程，发现 M 个断点已修复。看看行程安排合不合理？没问题说'阶段 C'，我来做预算和风险排查。"

## 阶段 C：审查排雷（花得值吗？有没有坑？）

> 用户说"阶段 C"时执行。先读 trip-spec.md 中阶段 B 的内容（特别是逐日行程表），然后写入「阶段 C」章节。

### 做什么

**核心问题：预算合理吗？有什么风险？怎么优化？**

1. **算账**：整体预算估算
2. **排雷**：风险识别与应对
3. **优化**：行程和预算的优化建议

| 景点门票 | ¥xxx - xxx | ... |

| 购物 + 手信 | ¥xxx - xxx | 丰俭由人 |

### C2. 风险排查

对行程做风险扫描，重点关注：

| 审查维度 | 关注什么 | 典型的坑 |
|----------|----------|----------|
| **天气风险** | 当季天气对行程的影响 | 雨季户外行程全泡汤 |
| **交通风险** | 航班延误、交通罢工、堵车 | 最后一天赶不上飞机 |
| **健康风险** | 水土不服、中暑、蚊虫 | 吃坏肚子毁掉后半程 |
| **安全风险** | 治安、诈骗、宰客 | 被出租车绕路、被景区黑导游坑 |
| **政策风险** | 签证变化、入境新规 | 到了才发现政策变了 |
| **体力风险** | 行程太密、连续暴走 | 第三天就累垮 |

```markdown

| ... | ... | ... |
```

按类别列出需要准备的物品：

|------|------|
| 证件 | ... |
| 支付 | ... |
| App | ... |
| 防晒 | ... |
| 药品 | ... |
| 衣物 | ... |
| 电源 | ... |

### C4. 实施路径 — 倒排时间表

从"现在"到"出发日"的待办事项，按优先级排列：

| 时间节点 | 优先级 | 待办事项 |

| 即刻 | P0 | 订机票（越早越便宜） |

- **省钱技巧**：哪里可以省、怎么省

## 注意事项

- **不代订任何机票酒店**。只往 trip-spec.md 写规划文档。
- **两层分析不要混**。阶段 B 只管"行程走不走得通"，阶段 C 才管"花得值不值、有什么坑"。
- **用大白话**。目的地术语第一次出现时翻译 + 类比（比如"BTS 就是曼谷的高架地铁，跟深圳地铁差不多"）。
- **阶段 A 重点关注签证和天气**。签证不过一切白搭，天气决定能玩什么。
- **阶段 B 重点关注交通衔接**。大部分"计划很美好但执行崩了"的旅行，问题出在交通衔接上——两个景点之间怎么去？要多久？有没有车？
- **阶段 C 重点关注体力节奏**。大部分"前半程开心后半程煎熬"的旅行，问题出在体力分配上——每天走多少、有没有休息缓冲。
- **信息要有来源**。搜索获取的信息标注来源，价格标注参考时间。不确定的标"待确认"。
- **方案要可执行**。"找个好酒店"不是方案，"Asoke 区域、BTS 站步行 5 分钟内、300-500 元/晚的四星酒店，推荐 xxx"才是。
- **每个阶段写完立刻写入 trip-spec.md**。不要在对话中输出完整内容再写文件——直接写文件，对话中只给摘要。
