# Use SuperPower 命令

> **命令类型**：`phase`
>
> **所属主流程**：`goal-code-loop`
>
> **阶段定位**：`goal-code-loop` 的前置提醒阶段，用于在进入指定编码步骤前，先注入注意事项，再调用对应的 SuperPower 技能

## 定位

`use-superpower` 不是自动拦截器，也不直接替代后续技能。

它的职责是：

1. 由用户显式指定当前要进入的编码阶段
2. 读取该阶段预埋的注意事项
3. 在进入对应 SuperPower 技能前，先把这些约束说清楚
4. 明确哪些动作应该做、哪些动作不应机械执行

## Trigger

- `use-superpower`
- “写计划前先读 SuperPower 注意事项”
- “开始 SubAgent 开发前先读 SuperPower 注意事项”
- “现在进入写计划阶段，先走 use-superpower”
- “现在进入 Sub-agent 阶段，先走 use-superpower”
- “sub-agent”
- “subagent”

## 输入

这个命令要求用户显式指定当前阶段。

第一版支持两个阶段：

- `write-plan`
- `sub-agent`
- `subagent`（兼容写法）

如果没有明确阶段，不要自己猜，先追问当前要进入哪个步骤。标准阶段名使用 `sub-agent`，`subagent` 只作为兼容写法。

## 阶段 1：`write-plan`

适用于：

- 用户已经明确要进入写计划阶段
- 接下来要调用 `writing-plans`
- 需要先把“计划应该怎么写”说清楚

### 进入该阶段前必须强调

1. 计划不是泛泛说明文，必须写成：

```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

2. 每一步都要尽量落到真实文件、真实命令、真实验收，而不是“后续补充”。
3. 如果后续会依赖 `overpass`、`local-test`、`cds`、`tcc`，要在计划里提前写出来。
4. 不要把 `sh build.sh` 机械塞进每一步。
5. 要区分：
   - 当前只是写计划 / 读代码 / 明确边界
   - 当前真的涉及代码、生成物、依赖或编译前置条件变化

只有后者才值得在计划里安排重新 `sh build.sh`。

### 后续应进入的技能

- `writing-plans`

## 阶段 2：`sub-agent`

适用于：

- 计划已经写完
- 用户明确要进入 SubAgent 开发阶段
- 接下来要调用 SubAgent 执行型技能

### 进入该阶段前必须强调

1. 先明确当前任务的成功标准，不要直接开写。
2. SubAgent 开发时可以维持红绿灯测试思路，但不要机械地每轮都执行 `sh build.sh`。
3. 只有在这些情况出现时，才重新执行 `sh build.sh`：
   - 代码发生变化，且需要重新编译验证
   - 生成物发生变化
   - 依赖版本发生变化
   - 编译前置条件发生变化
4. 如果本轮没有改代码，优先复用上一次构建结果，不要重复全量 build。
5. 先最小修改，再最小验证，不要一上来跑最重的链路。
6. 如果当前阻塞点是下游依赖缺口，应先走 `overpass`，而不是在本仓写临时兼容代码绕过去。

### 后续应进入的技能

- `subagent-driven-development`
- `executing-plans`
- 需要本地验证时，再结合 `local-test`

## 执行规则

1. 先判断用户显式指定的是哪个阶段。
2. 输出该阶段的注意事项。
3. 明确下一步应该进入哪个技能。
4. 本命令本身不替代后续技能执行。

## 收口规则

只有同时满足以下条件，才算本命令完成：

- 已明确当前阶段
- 已输出对应阶段的注意事项
- 已指出下一步应调用的技能
- 已说明本命令只负责前置提醒，不直接替代后续技能
