# RPC Request Shape Check Command

> **命令类型**：`shared`
>
> **典型调用方**：`../workflows/goal-rpc-loop.md`、`../phases/rpc-goal-clarify.md`
>
> **参考证据**：`../../references/evidence/2026-04-19-doushan-rpc-shape-and-scm-sha/index.md`

## Trigger

- `rpc-request-shape-check`
- `request shape check`
- “先帮我确认这个 RPC 请求是不是客户端上报态”
- “按代码看看这个字段到底该怎么构造”
- “先别怀疑服务端逻辑，先排查请求形态对不对”
- “为什么同样的字段放进请求里还是被前置校验拦掉”

## 目标

在真正怀疑服务端逻辑、版本或配置之前，先回答这三个问题：

1. 服务端真正消费了哪些请求字段
2. 当前请求体是否符合客户端真实上报态，而不是服务端存储态
3. 哪些字段会影响前置分支命中，需要优先对齐后再复测

## 适用场景

- 真实 RPC 已经能打通，但结果和预期不一致，怀疑请求体形态有误
- 代码里同时存在“客户端上报态”和“服务端存储态”两套相似结构，容易直接抄错
- 某些前置校验在进入核心逻辑前就把请求拦掉，导致误判成“服务没生效”
- 用户给了宏观验收预期，但还没把请求字段推导到可执行粒度

不适用于：

- 目标方法和关键请求字段都已经通过真实调用验证过
- 当前问题已经确认是环境、配置、部署版本或实例流量问题
- 还没定位到目标 RPC，连 method 和最小入参都不清楚

## 固定检查链路

默认按下面顺序做，不要跳成“先看回包不对，再直接怀疑服务端”：

1. 先读代码，确认服务端真正消费哪些字段
2. 标记会影响前置判断的字段
3. 判断当前 JSON 是客户端上报态还是服务端存储态
4. 只修正会影响分支命中的字段，保留其余骨架不动
5. 用修正后的请求重新做一轮真实 RPC
6. 再根据新结果决定是否进入日志、版本或配置排查

## 第 1 步：先读代码，不凭字段名想当然

至少回答以下问题：

- handler / service 最终读取的是请求里的哪个字段
- 请求透传字段和服务端组装字段，最终参与判断的是哪一个
- 有没有前置一致性校验、空值保护或 shape normalizer
- 哪些字段只用于展示，哪些字段真正决定分支

如果读代码后发现服务端根本不看某个字段，就不要继续围绕它调参。

## 第 2 步：找出前置分支敏感字段

优先标记以下几类字段：

- 会影响一致性校验的字段
- 会影响 sign / version / completed 判断的字段
- 会影响资源集合比对的字段
- 会改变服务端是否进入后续逻辑的字段

重点不是“把请求补完整”，而是“先让请求通过关键前置判断”。

## 第 3 步：区分客户端上报态与服务端存储态

如果同一个结构既会出现在客户端请求，也会出现在服务端存储或回包中，必须显式判断当前手上的 JSON 属于哪一类。

常见判断方式：

- 客户端上报态通常只保留服务端实际消费的最小必要字段
- 服务端存储态经常带有生成中、补充态、落库态字段，不适合直接原样回灌
- 回包里出现的完整对象，不代表下一次请求就应该原样透传

针对本次已验证案例，`Extra.widget` 的经验是：

- 需要保留 `completed`、`version`、`dou_shan_actual_sign`
- 需要保证 `image_id` 等一致性字段与当前状态匹配
- 不应把 `producing` 等服务端存储态字段直接照抄进下一次请求

## 第 4 步：按最小修改原则修正请求

修正请求时遵守：

- 先复用最近一轮已跑过的请求骨架
- 只改会影响前置校验或核心分支命中的字段
- 不顺手重排整个 JSON，也不把全部字段重写一遍
- 新请求体必须另存为新文件，避免覆盖历史证据

优先参考：

- `../../references/recent-command-reuse.md`
- 最近 round 的 `requests/`、`responses/`、`verdict.md`

## 第 5 步：复测并做结果判定

复测后先判断：

1. 是否已经通过原先的前置校验
2. 是否进入了新的逻辑分支
3. 回包差异是否说明请求形态问题已关闭

如果修正请求形态后结果明显变化，优先把结论写成“原问题主要是请求构造错误”，不要继续把锅甩给版本或逻辑。

如果修正后仍完全无变化，再进入：

- `rpc-pod-triage`
- 版本 / SCM SHA 核对
- 配置或 TCC 排查

## 建议落盘产物

至少保留：

- 本轮代码定位结论
- 旧请求与新请求路径
- 影响前置判断的关键字段清单
- 新回包路径
- 一句话结论

推荐落到当前 session 目录下的分析或 round 文档中，而不是只停留在口头总结。

## 收口规则

只有同时满足下面条件，才算本命令完成：

- 已说明服务端真正消费哪些字段
- 已判断当前请求属于哪种 shape
- 已指出需要改的关键字段，而不是泛泛地说“参数可能不对”
- 已基于修正后的请求做过至少一轮真实 RPC 或明确说明为何暂不能复测
- 已给出下一步建议：继续 RPC 复测，还是转入日志 / 版本 / 配置排查
