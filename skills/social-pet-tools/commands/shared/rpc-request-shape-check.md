# RPC Request Shape Check Command

> **命令类型**：`shared`
>
> **典型调用方**：`../workflows/goal-rpc-loop.md`、`../phases/rpc-goal-clarify.md`
>
> **参考资料**：`../../references/goal-rpc-loop-controller-contract.md`、`../../references/evidence/2026-04-19-doushan-rpc-shape-and-scm-sha/index.md`

## Trigger

- `rpc-request-shape-check`
- `request shape check`
- “先帮我确认这个 RPC 请求是不是客户端上报态”
- “按代码看看这个字段到底该怎么构造”
- “先别怀疑服务端逻辑，先排查请求形态对不对”
- “为什么同样的字段放进请求里还是被前置校验拦掉”

## 定位

这是 `goal-rpc-loop` 借用的 `shared worker`，只负责回答“当前阻塞点是不是请求形态错误”。

- 它不负责替主流程决定下一阶段
- 它不直接宣布“去查日志”或“去查版本”
- 它的完成标准是把请求 shape 问题收敛成可复测、可回到主流程判断的证据

执行完后，控制权必须回到 `goal-rpc-loop` controller，由主流程决定：

- 回到 `rpc-first-pass`
- 继续 `rpc-gap-loop`
- 还是进入 `rpc-pod-triage` / 版本 / 配置排查

## 目标

在真正怀疑服务端逻辑、实例日志、版本或配置之前，先回答下面三个问题：

1. 服务端真正消费了哪些请求字段
2. 当前请求体是客户端上报态，还是服务端存储态 / 回包态
3. 哪些字段会影响前置校验或关键分支命中，应该先最小化修正后复测

## 何时调用

- 真实 RPC 已经能打通，但结果和预期不一致，且差异看起来发生在核心逻辑之前
- 代码里同时存在“客户端上报态”和“服务端存储态”两套相似结构，容易把回包对象直接回灌成请求
- 某些前置校验在进入核心逻辑前就把请求拦掉，导致误判成“服务没生效”
- goal-rpc-loop 当前阻塞点明确是“请求形态是否正确”，而不是“实例有没有收到”或“版本有没有更新”

## 何时不要调用

- 目标方法和关键请求字段都已经通过真实调用验证过
- 请求已经通过代码和真实调用验证为正确 shape，当前问题明显是日志、版本、配置、流量或实例侧问题
- 还没定位到目标 RPC，连 method、最小入参和目标对象都不清楚

此时更合适的动作通常是：

- 先走 rpc-goal-clarify
- 或由主流程直接派 rpc-pod-triage
- 或在已确认非请求问题后，再做版本 / SCM SHA / CDS / TCC 排查

## 不要直接跳版本 / 日志排查

当当前阻塞点仍是“请求 shape 是否正确”时，先做本命令，不要直接跳到 rpc-pod-triage、版本 / SCM SHA、配置 / TCC。只有在修正 shape 后仍完全无变化，或已确认请求已经命中正确前置分支时，才交回主流程继续向外分流。

## 输入

- 明确的目标 RPC method
- 最近一轮真实请求体或最小请求骨架
- 最近一轮真实回包或前置报错信息
- 相关 handler / service / validator / normalizer 代码位置
- 当前预期行为与实际差异

## 固定检查链路

默认按下面顺序做，不要跳成“回包不对 -> 直接查日志 / 版本 / 配置”：

1. 先读代码，确认服务端真正消费哪些字段
2. 标记会影响前置判断的字段
3. 判断当前 JSON 是客户端上报态还是服务端存储态
4. 按最小修改原则修正请求，不重写整个骨架
5. 用修正后的请求做一轮真实 RPC 复测
6. 基于复测结果，把控制权交回主流程决定下一跳

## 第 1 步：先读代码，不凭字段名想当然

至少回答以下问题：

- handler / service 最终读取的是请求里的哪个字段
- 请求透传字段和服务端组装字段，最终参与判断的是哪一个
- 有没有前置一致性校验、空值保护或 shape normalizer
- 哪些字段只用于展示，哪些字段真正决定分支

如果读代码后发现服务端根本不看某个字段，就不要继续围绕它调参。

## 第 2 步：找出前置分支敏感字段

优先标记以下几类字段：

- 会影响一致性校验的字段
- 会影响 sign / version / completed 判断的字段
- 会影响资源集合比对的字段
- 会改变服务端是否进入后续逻辑的字段

重点不是“把请求补完整”，而是“先让请求通过关键前置判断”。

## 第 3 步：区分客户端上报态与服务端存储态

如果同一个结构既会出现在客户端请求，也会出现在服务端存储或回包中，必须显式判断当前手上的 JSON 属于哪一类。

常见判断方式：

- 客户端上报态通常只保留服务端实际消费的最小必要字段
- 服务端存储态经常带有生成中、补充态、落库态字段，不适合直接原样回灌
- 回包里出现的完整对象，不代表下一次请求就应该原样透传

针对本次已验证案例，`Extra.widget` 的经验是：

- 需要保留 `completed`、`version`、`dou_shan_actual_sign`
- 需要保证 `image_id` 等一致性字段与当前状态匹配
- 不应把 `producing` 等服务端存储态字段直接照抄进下一次请求

## 第 4 步：按最小修改原则修正请求

修正请求时遵守：

- 先复用最近一轮已跑过的请求骨架
- 只改会影响前置校验或核心分支命中的字段
- 不顺手重排整个 JSON，也不把全部字段重写一遍
- 新请求体必须另存为新文件，避免覆盖历史证据

优先参考：

- `../../references/recent-command-reuse.md`
- 最近 round 的 `requests/`、`responses/`、`verdict.md`

## 第 5 步：复测并做结果判定

复测后先判断：

1. 是否已经通过原先的前置校验
2. 是否进入了新的逻辑分支
3. 回包差异是否说明请求形态问题已经被关闭

如果修正请求 shape 后结果明显变化，优先写明“原问题主要来自请求构造错误”，不要继续先入为主地怀疑版本或逻辑。

如果修正后仍完全无变化，再把结论整理给主流程，由 controller 决定是否进入：

- `rpc-pod-triage`
- 版本 / SCM SHA 核对
- 配置或 TCC 排查

如果还没法复测，也要明确卡点是什么，避免主流程误以为 shape 问题已经排除。

## 建议落盘产物

至少保留：

- 代码定位结论：服务端真正消费哪些字段
- 旧请求与新请求路径
- 关键敏感字段清单：哪些字段影响前置判断，为什么
- 新回包路径或“暂不能复测”的明确原因
- 一句话结论
- 下一步建议

推荐落到当前 session 的分析或 round 文档中，至少能被主流程重新读取和归并。

## 返回给主流程的最少信息

- 当前阻塞点是否已被收敛为请求 shape 问题
- 是否已经产出可复用的新请求文件
- 复测是否执行过，结果是否发生变化
- 下一步更推荐回到哪条路：继续 RPC 复测，还是转日志 / 版本 / 配置排查

注意：这里给的是建议，不是替 controller 做最终决策。

## 收口规则

只有同时满足下面条件，才算本命令完成：

- 已说明服务端真正消费哪些字段
- 已判断当前请求属于哪种 shape，而不是泛泛地说“参数可能不对”
- 已指出需要改的关键字段，并解释它们为什么影响前置判断
- 已基于修正后的请求做过至少一轮真实 RPC，或明确说明为何暂不能复测
- 已留下可归并的产物路径与一句话结论
- 已给出明确下一步建议，并把控制权交回主流程
