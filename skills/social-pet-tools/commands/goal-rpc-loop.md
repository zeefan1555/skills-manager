# Goal RPC Loop 命令

> **引用规范**：`../references/conventions.md`、`../references/mapping.md`、`../references/goal-rpc-loop-example.md`、旧 skill `social-pet-rpc-verify-loop`

## Trigger

- `goal-rpc-loop`
- “我给你一份宏观 RPC 预期文档，你帮我一步步推导请求再验证”
- “先找到请求，再打 RPC，不对就继续循环调试”
- “按目标驱动做 RPC 验证闭环”
- “我只知道大概预期输入输出，你帮我从零开始构造请求”

## 目标

在 `social_pet` 仓库中，把一份较粗的 RPC 预期文档，变成一套可执行、可验证、可循环的验证闭环：

1. 先收敛高层预期和成功标准
2. 再推导接口链路与最小请求体
3. 串行执行真实 RPC
4. 对返回做关键字段断言
5. 如果结果不对，继续做请求 / 状态 / 配置 / 语义层面的下一轮验证
6. 最后给出事实、证据路径和收敛结论

## 输入

优先收集这些输入：

- 用户给出的原始预期文档或口头说明
- 目标场景：新增用户 / 存量用户 / 指定会话 / 指定设备
- 需要验证的接口链路，哪怕一开始只是大概猜测
- 环境参数
- 当前已知的预期字段

如果输入仍然很粗，先补齐：

- 要验证的是哪个链路
- 每一步的成功标准是什么
- 当前是否已有请求模板或历史证据

## 固定约束

- 不要一上来就“先试个请求看看”
- 先把任务改写成可验证目标，再执行
- 多步骤任务必须先给出简短计划，格式固定为：

```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

- 每次 RPC 执行都要保留请求、返回、`log_id` 和当前断言结果
- 如果结果不对，不要立刻判 bug，先分类原因
- 这个命令是 Agent 级别的编排入口，可以在循环过程中动态调用其他命令

## 动态依赖命令

在循环过程中，按当前事实动态选择下面这些命令：

| 当前需求 | 优先命令 | 说明 |
|---|---|---|
| 需要把较粗预期先整理成请求/返回/断言文档，但还不实际打 RPC | `rpc-verify-spec` | 先产出结构化验证文档 |
| 真实 RPC 返回不对，需要按 `log_id -> pod -> instance log` 深挖 | `rpc-pod-triage` | 走实例日志链路排障 |
| 发现问题来自 `WidgetCfg.xlsx` / CDS / Excel 配置 | `cds` | 修改、校验并提交配置表 |
| 发现问题来自 TCC / 白名单 / 实验开关 | `tcc` | 修改并发布 TCC |
| 已经改了代码，需要继续做“本地测试 + RPC 重试” | `rpc-verify` | 进入修复后复测闭环 |

判断原则：

1. 先看当前差异属于请求、状态、环境、配置还是语义问题
2. 再选择最短路径命令
3. 每次只补当前 loop 需要的动作，不同时展开所有命令

## 执行步骤

### 第 1 步：记录原始预期

不要把用户最开始的高层预期直接当成最终断言。

先记录：

- 宏观输入是什么
- 宏观输出是什么
- 哪几个接口可能相关
- 当前最初预期是什么

### 第 2 步：先收敛成功标准

把模糊任务改写成强成功标准。

例如：

- “帮我验证这个 RPC”
  - 改写成：
  - “构造最小请求，调用 RPC，并验证关键字段 `X` 是否等于 `Y`”
- “按验收文档跑一轮”
  - 改写成：
  - “先拆接口链路，再逐步执行，每一步都带断言”

### 第 3 步：推导接口链路与请求

从下面几类信息推导请求：

1. 用户原始预期
2. 代码实现
3. 状态依赖
4. 环境 / 配置依赖

每个请求都要回答：

- 为什么是这个接口
- 为什么是这个顺序
- 哪些字段是必填
- 哪些字段影响分支
- 哪些字段必须和前一步一致

优先最小请求体，不要一开始就抄整份客户端原包。

### 第 4 步：串行执行 RPC 并保存证据

每一步都要保存：

- `NN-req-<method>.json`
- `NN-resp-<method>.json`
- `log_id`
- 当前断言结果

### 第 5 步：判断当前结果

按 3 层断言判断：

1. 协议层：接口是否成功
2. 业务层：关键字段是否符合当前预期
3. 链路层：当前结果是否支撑继续执行下一步

### 第 6 步：结果不对时进入下一轮

先分类差异：

1. 请求问题
2. 状态问题
3. 环境 / 配置问题
4. 语义问题

然后：

- 如果需要日志深挖：转 `rpc-pod-triage`
- 如果需要配置修改：转 `cds` 或 `tcc`
- 如果需要改码后继续重跑：转 `rpc-verify`

如果只是某一步有问题，就只重跑必要步骤，不要整条链全部重来。

## 报告与证据

产物默认落盘到当前 session 的 `rpc-verify/` 子目录（详见 `../references/conventions.md`）：

```
docs/social-pet/<date-topic>/rpc-verify/
```

每轮至少保留：

- 请求体
- 回包
- `log_id`
- 当前判定：`PASS / FAIL / EXPECTED_DIFF / UNKNOWN`
- 下一轮动作

## 何时回退到旧 skill

遇到下面情况，回退到旧 skill `social-pet-rpc-verify-loop`：

- 需要更完整的历史 RPC 回溯方法
- 需要复用旧 skill 内部更深的 `rpc-call` 参数模板
- 当前聚合层命令不足以覆盖细节

## 推荐返回结构

- 用户原始预期是什么
- 当前收敛出的成功标准是什么
- 当前接口链路与请求体是什么
- 每轮 RPC 的实际结果是什么
- 当前结论与下一轮动作是什么
- 当前是否触发了其他命令，以及为什么触发

## 常见错误

- 把用户最开始的高层预期直接当成最终断言
- 请求体一开始就过重，导致难以排查
- 返回不对就立刻叫做 bug
- 没保存请求和回包，只凭终端滚屏总结
- 没有写清楚下一轮要改什么
- 明明应该切去 `rpc-pod-triage` / `cds` / `tcc`，却还在当前命令里硬查

## 收口规则

只有同时满足以下条件，才算本命令完成：

- 已把高层预期改写成强成功标准
- 已完成至少一轮真实 RPC 验证并保存证据
- 如果结果不对，已明确进入下一轮的原因和改动方向
- 已说明当前是否需要切换到其他命令
- 最终已说明当前是通过、失败、预期差异，还是仍待继续验证
