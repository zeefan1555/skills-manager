# 03. Loop 2：发现不对后继续循环调试

## 为什么进入 Loop 2

在 Loop 1 里，`GetWidget` 的结果和最初直觉不一致：

- 只出现了 `401/402`
- 没有 `403/404/405`

这时不能直接下结论说接口错了。

要继续做的是：

1. 查代码
2. 查配置
3. 查运行时状态
4. 再决定这是不是 bug

## Loop 2 做了什么

### 1. 先查当前运行时 subtype

继续看 `GetWidget` 返回和摘要后，发现：

- 当前 `widget_sub_type=1`

### 2. 再查当前实际下发的资源键

发现：

- `extra_resource_keys=["401","402"]`

### 3. 再查配置

配置摘录显示：

- `401 -> subType=1`
- `402 -> subType=1`
- `403 -> subType=2`
- `404 -> subType=2`
- `405 -> subType/extraParam 为空`

也就是说：

1. `403/404` 根本不是当前 `subType=1` 场景的资源
2. `405` 当前不具备被下发条件

## Loop 2 的判断

到了这里，问题就不再是“接口是不是错了”，而变成：

- 最初的宏观预期太粗
- 需要根据代码和配置把预期修正为“当前场景只出现 `401/402`”

所以这轮的判定是：

- `GetWidget` 不是 `FAIL`
- 而是 `EXPECTED_DIFF`

## Loop 2 的输出

经过这轮调试后，验收口径被收敛成：

1. 主链路是通的
2. `GetWidget` 在当前场景只返回 `401/402`
3. `403/404` 只在 `subType=2`
4. `405` 无有效 `subType/extraParam`

这时才可以进入最后的结论整理。
