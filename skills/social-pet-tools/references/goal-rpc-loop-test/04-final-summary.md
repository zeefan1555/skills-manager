# 04. 最终总结

## 这次实验最后得到的结论

### 主链路

`GetPetElfCfg -> UpdateWidget -> GetWidget -> GetPetElfCfg`

在存量用户和新增用户两个场景下，主链路都跑通了。

### 关键判断

1. 第一次 `GetPetElfCfg` 返回 `NeedUpdateWidget=true`
2. `UpdateWidget` 返回成功
3. 第二次 `GetPetElfCfg` 返回 `NeedUpdateWidget=false`
4. `GetWidget` 在当前场景下只返回 `401/402`

## 为什么这不是一开始就能写出来的

因为用户最开始给的是宏观预期，不是完整请求和完整断言。

真正的结论是通过下面两轮 loop 一步步收敛出来的：

1. 先跑主链路，发现 `GetWidget` 和直觉不一致
2. 再查代码 / 配置 / subtype，确认这是配置语义导致的正常结果

## 这次实验真正要复用的不是“结果”，而是“方法”

下次拿到新的宏观预期时，可以继续按这套流程：

1. 先记录原始预期
2. 再找接口顺序
3. 再查代码推导请求
4. 发第一轮请求
5. 不符合预期就进入下一轮 loop
6. 最后再整理总结

## 这次实验的证据目录

- 存量用户：`artifacts/stock/`
- 新增用户：`artifacts/new/`

## 这次实验最值得保留的经验

当“结果和最初预期不一致”时，不能直接判失败。

应该继续问：

1. 是请求没构造对
2. 是状态没准备对
3. 是环境没命中
4. 还是代码 / 配置语义本来就是这样

只有把这层差异解释清楚，loop 才算真正闭环。
