# Flight Scout：机票价格侦察

## 这个 skill 做什么

用户告诉你"从哪出发、去哪、什么时候、几个人"，你分三步帮他搜集和对比机票价格：

1. **阶段 A**：从携程搜集出发城市直飞方案 → 存入 `flight-a-ctrip.md`
2. **阶段 B**：从 Trip.com 搜集中转城市方案（含陆路交通） → 存入 `flight-b-trip.md`
3. **阶段 C**：汇总两份文件，综合对比 → 存入 `flight-c-compare.md`

**不代订机票**，只产出分析文档。

## 为什么要分步

一口气搜完所有组合再输出，上下文太长容易乱。分步走的好处：
- 每一步搜完存档，用户可以先审阅确认，再进入下一步
- 阶段 C 对比时读两份已有 MD，信息干净不混杂
- 如果某个阶段搜的价格不对，可以单独重跑，不影响其他阶段

## 工作模式

```
用户给出目标（出发地、目的地、日期、人数）
      |
用户说"阶段 A"  →  搜携程（出发城市直飞） → 写 flight-a-ctrip.md
      |
用户审阅 → 确认或修正
      |
用户说"阶段 B"  →  搜 Trip.com（中转城市+陆路交通） → 写 flight-b-trip.md
      |
用户审阅 → 确认或修正
      |
用户说"阶段 C"  →  读 A + B 两份文件 → 综合对比 → 写 flight-c-compare.md
      |
完成，用户拿着结论去下单
```

### 关键规则

- **用户触发**：用户说"阶段 A/B/C"时才执行对应阶段，不自动推进
- **分文件存档**：每个阶段写独立 MD 文件，不合并
- **阶段间依赖**：C 必须读 A 和 B 已写入的文件
- **可重跑**：用户对某阶段不满意，重跑时覆盖对应 MD
- **快速模式**：如果用户说"直接全部跑完"或"一次性搞定"，则依次执行 A→B→C

## 首次触发：确认信息

用户首次触发时，从对话上下文提取以下信息，提取不到的才追问：

| 信息 | 必须 | 示例 |
|------|------|------|
| 出发城市 | ✅ | 深圳 |
| 目的城市 | ✅ | 曼谷 |
| 去程日期 + 大致时间偏好 | ✅ | 4月30日下午5点半左右出发 |
| 回程日期 + 时间偏好 | ✅ | 5月7日，不要太晚（第二天上班） |
| 人数 | ✅ | 2人 |
| 中转城市 | 可选 | 香港（默认根据出发地推导） |
| 其他偏好 | 可选 | 不要红眼、要直飞、行李多等 |

确认完毕后告诉用户："信息已确认。准备好了说'阶段 A'，我先去携程搜深圳直飞的方案。"

### 中转城市推导

如果用户没有指定中转城市，根据出发地自动推导：

| 出发城市 | 推荐中转 | 理由 |
|---------|---------|------|
| 深圳 | 香港 HKG | 大湾区国际枢纽，廉航多，常有特价 |
| 上海 | — | 浦东本身就是大枢纽，一般不需要绕 |
| 北京 | — | 首都/大兴本身航线丰富 |
| 成都 | 昆明 KMG | 东南亚方向昆明有价格优势 |
| 其他内陆城市 | 广州 CAN / 香港 HKG | 南方枢纽中转东南亚 |

不是所有出发地都需要查中转——如果出发地本身国际航线丰富，诚实告诉用户"直飞大概率最优，不用绕"。

## 阶段 A：携程搜集 — 出发城市直飞方案

> 用户说"阶段 A"时执行。完成后写入 `flight-a-ctrip.md`。

### 做什么

从**携程（ctrip.com / trip.com 国内版）** 搜集出发城市直飞目的地的所有方案。

### A1. 搜索内容

对以下每种组合分别搜索价格：

| 编号 | 购票方式 | 说明 |
|------|---------|------|
| A-1 | **往返票** | 出发城市 ↔ 目的城市，一次买好去程+回程 |
| A-2 | **去程单程** | 出发城市 → 目的城市，只买去程 |
| A-3 | **回程单程** | 目的城市 → 出发城市，只买回程 |
| A-4 | **单程拼接总价** | A-2 + A-3 的总价（对比 A-1 看哪个划算） |

### A2. 搜索方法

使用网络搜索工具，搜索关键词模板：

```
携程 {出发城市}飞{目的城市} {去程日期} 机票
携程 {出发城市}到{目的城市} {去程日期} 单程机票
携程 {目的城市}飞{出发城市} {回程日期} 单程机票
携程 {出发城市}{目的城市} {去程月份} 往返机票
```

如果搜索结果中能直接获取价格信息就提取，否则尝试访问携程的航班搜索页面获取更详细的数据。

### A3. 每个方案需要记录的信息

| 字段 | 说明 |
|------|------|
| 航空公司 | 如南方航空、深圳航空、亚航等 |
| 航班时间 | 出发时间 → 到达时间（注意时差） |
| 飞行时长 | 如 3h |
| 到达机场 | BKK 素万那普 or DMK 廊曼（影响后续交通） |
| 票价 | 含税价，人民币 ¥，标注是否为每人价格 |
| 行李额 | 免费托运额度（廉航通常无免费行李） |
| 退改政策 | 可退/不可退/退改费多少 |

### A4. 产出格式

写入 `flight-a-ctrip.md`：

```markdown

| 航空公司 | 去程航班 | 回程航班 | 含税总价（{人数}人） | 单人价 | 行李额 | 退改 |
|---------|---------|---------|-------------------|--------|--------|------|
| ... | {时间} {机场} | {时间} {机场} | ¥xxx | ¥xxx | ... | ... |

## A-2. 去程单程（{出发城市} → {目的城市}）

| 航空公司 | 航班时间 | 到达机场 | 含税单人价 | 行李额 |
|---------|---------|---------|----------|--------|
| ... | ... | ... | ¥xxx | ... |

## A-3. 回程单程（{目的城市} → {出发城市}）

| 航空公司 | 航班时间 | 到达机场 | 含税单人价 | 行李额 |
|---------|---------|---------|----------|--------|
| ... | ... | ... | ¥xxx | ... |

## A-4. 单程拼接 vs 往返票对比

| 方案 | 去程 | 回程 | 总价（{人数}人） | 对比往返票 |
|------|------|------|----------------|-----------|
| 最便宜往返票 | ... | ... | ¥xxx | 基准 |
| 最便宜单程拼接 | A-2 最低 + A-3 最低 | ... | ¥xxx | 贵/便宜 ¥xxx |

## 小结
{2-3 句话总结：携程渠道下，往返票和单程拼接哪个更划算，最低价是多少}
```

## 阶段 B：Trip.com 搜集 — 中转城市方案（含陆路交通）

> 用户说"阶段 B"时执行。完成后写入 `flight-b-trip.md`。

### 做什么

从 **Trip.com（国际版）** 搜集经中转城市出发的方案，并加上"出发城市→中转城市"的陆路交通成本。

以"深圳→香港→曼谷"为例，这个阶段要搜集两部分：
1. **陆路交通**：深圳到香港机场怎么去、多少钱、多久
2. **香港出发的机票**：香港飞曼谷的往返票和单程票

### B1. 陆路交通搜集

搜索"出发城市→中转城市机场"的所有合理交通方式：

**搜索关键词模板**：
```
{出发城市}到{中转城市}机场 交通方式 2026
{出发城市}到{中转城市}机场 最便宜 怎么去
蛇口码头到香港机场 船票 价格
深圳湾口岸到香港机场 大巴
```

**需要记录的信息**：

| 字段 | 说明 |
|------|------|
| 交通方式 | 如蛇口码头快船、深圳湾大巴+机场快线、高铁+机场快线等 |
| 出发点 | 如蛇口码头、深圳湾口岸、福田口岸 |
| 耗时 | 含通关时间 |
| 费用 | 单人单程 |
| 班次 | 是否需要提前购票、班次频率 |
| 注意事项 | 如需港澳通行证、需提前值机等 |

### B2. 中转城市机票搜索

和阶段 A 一样的结构，但出发地换成中转城市，搜索平台用 Trip.com：

| 编号 | 购票方式 | 说明 |
|------|---------|------|
| B-1 | **往返票** | 中转城市 ↔ 目的城市 |
| B-2 | **去程单程** | 中转城市 → 目的城市 |
| B-3 | **回程单程** | 目的城市 → 中转城市 |
| B-4 | **单程拼接总价** | B-2 + B-3 |

**搜索关键词模板**：
```
Trip.com {中转城市} to {目的城市} {日期} flight
Trip.com {中转城市}飞{目的城市} {日期}
site:trip.com {中转城市} {目的城市} flight {月份}
```

### B3. 计算含陆路的总成本

这是阶段 B 的关键——把陆路交通费加进机票价格，算出"门到门"的真实总成本：

```
方案总成本 = 陆路交通（去程+回程）× 人数 + 机票价格 × 人数
```

注意：
- 陆路交通是**往返**的（去的时候要从出发城市到中转城市，回来的时候要从中转城市回出发城市）
- 如果回程航班选择直接飞回出发城市（混合方案），回程就没有陆路交通

写入 `flight-b-trip.md`：

```markdown

> 搜索时间：{当前日期时间}

> 陆路路线：{出发城市} → {中转城市}机场

## B0. 陆路交通：{出发城市} → {中转城市}机场

| 方式 | 路线 | 耗时 | 单人单程费用 | 班次 | 注意事项 |

|---------|---------|---------|-------------------|--------|--------|
| ... | ... | ... | ¥xxx | ¥xxx | ... |

## B-2. 去程单程（{中转城市} → {目的城市}）

## B-3. 回程单程（{目的城市} → {中转城市}）

| 航空公司 | 航班时间 | 到达机场 | 含税单人价 | 行李额 |
|---------|---------|---------|----------|--------|
| ... | ... | ... | ¥xxx | ... |

|------|----------------|----------------------|----------|
```

## 阶段 C：综合对比

> 用户说"阶段 C"时执行。先读 flight-a-ctrip.md 和 flight-b-trip.md，然后写入 `flight-c-compare.md`。

### 做什么

**把阶段 A 和阶段 B 的数据放在一起，算出所有方案的真实总成本，排序后给出最优推荐。**

### C1. 汇总所有方案

从两份 MD 中提取所有方案，统一格式对比。除了 A 和 B 各自内部的方案，还要考虑**混合方案**：

| 混合方案类型 | 说明 |
|------------|------|
| 去程直飞 + 回程经中转 | 去的时候图方便直飞，回来绕一下更便宜 |
| 去程经中转 + 回程直飞 | 去的时候绕一下便宜，回来直飞方便 |

混合方案的成本 = 去程机票 + 回程机票 + 涉及中转的那程陆路交通（单程×人数）。

## 注意事项

- **每个阶段写完立刻写入对应 MD 文件**。对话中只给摘要，不输出完整表格。
- **搜索为王**。每个价格都必须通过搜索获取，不凭记忆编造。搜不到的标"待确认"。
- **总成本思维**。永远比"总成本"——机票 + 陆路交通 + 行李费 + 隐性成本。
- **标注时效**。机票价格随时变，所有价格标注搜索时间，提醒用户"价格仅供参考，建议 24h 内下单"。
- **标注人数**。所有价格明确标注是"单人价"还是"X人总价"，避免混淆。
- **尊重用户约束**。用户的时间偏好（不要红眼、不要太晚）作为硬性筛选条件。
- **陆路交通要实用**。不只列价格，还要告诉用户具体怎么走、需要什么证件、在哪买票。
