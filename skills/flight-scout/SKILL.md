# Flight Scout：机票价格侦察

## 这个 skill 做什么

用户告诉你"从哪出发、去哪、什么时候、几个人"，你分三步帮他在官网查询并对比机票价格：

1. **阶段 A**：通过 `playwright-cli` 访问携程官网 `https://flights.ctrip.com/`，搜集出发城市直飞方案 → 存入 `flight-a-ctrip.md`
2. **阶段 B**：通过 `playwright-cli` 访问 Trip.com 官网 `https://www.trip.com/flights/`，搜集中转城市方案（含陆路交通） → 存入 `flight-b-trip.md`
3. **阶段 C**：汇总两份文件，综合对比 → 存入 `flight-c-compare.md`

**不代订机票**，只产出分析文档。

## 为什么要分步

一口气搜完所有组合再输出，上下文太长容易乱。分步走的好处：
- 每一步搜完存档，用户可以先审阅确认，再进入下一步
- 阶段 C 对比时读两份已有 MD，信息干净不混杂
- 如果某个阶段搜的价格不对，可以单独重跑，不影响其他阶段

## 官网直查原则

- **只查官网**：机票信息只允许来自官网真实结果页，不允许使用搜索引擎、聚合比价页、摘要页或二手转载页
- **必须像真人一样操作**：先读取 `playwright-cli` 技能 `/Users/bytedance/.skills-manager/skills/playwright-cli`，根据技能说明找到当前可用命令，再按该技能提示启动无头浏览器，模拟真人打开页面、输入条件、点击搜索、展开详情
- **结果必须和官网一致**：航班、价格、时间、机场、行李、退改等字段，只能记录页面实际展示值
- **抓不到就写待确认**：如果官网页面因登录、验证码、风控、地区限制或币种切换导致字段不稳定，必须写 `待确认`
- **禁止凭记忆补价**：任何价格都不能凭历史经验、聊天上下文或模型记忆补全

## 工作模式

```
用户给出目标（出发地、目的地、日期、人数）
      |
用户说"阶段 A"  →  用 Playwright 访问携程官网（出发城市直飞） → 写 flight-a-ctrip.md
      |
用户审阅 → 确认或修正
      |
用户说"阶段 B"  →  用 Playwright 访问 Trip.com 官网（中转城市+陆路交通） → 写 flight-b-trip.md
      |
用户审阅 → 确认或修正
      |
用户说"阶段 C"  →  读 A + B 两份文件 → 综合对比 → 写 flight-c-compare.md
      |
完成，用户拿着结论去下单
```

## Reference SOP

- 单程整页抓取：`reference/ctrip-oneway-full-page-sop.md`
- 往返选去程后抓返程全集：`reference/ctrip-roundtrip-outbound-to-return-sop.md`
- 往返第一页去程逐个迭代：`reference/ctrip-roundtrip-all-outbounds-iterative-sop.md`

这两个 reference 是最小原子化单位：

- **单程整页抓取**：解决“某个单程结果页的全部航班怎么一次抓全”
- **往返选去程后抓返程全集**：解决“先固定一个去程，再把它对应的全部返程组合抓全”
- **往返第一页去程逐个迭代**：解决“第一页去程全部跑完，并且每轮都把进度写回主表和 state，同时复用同一个 session”

后续更大的流程（例如所有去程逐个展开成 `N × N` 往返全集）应建立在这两个原子 SOP 之上。

### 关键规则

- **用户触发**：用户说"阶段 A/B/C"时才执行对应阶段，不自动推进
- **分文件存档**：每个阶段写独立 MD 文件，不合并
- **阶段间依赖**：C 必须读 A 和 B 已写入的文件
- **可重跑**：用户对某阶段不满意，重跑时覆盖对应 MD
- **快速模式**：如果用户说"直接全部跑完"或"一次性搞定"，则依次执行 A→B→C
- **官网优先**：阶段 A 只查 `https://flights.ctrip.com/`，阶段 B 只查 `https://www.trip.com/flights/`
- **浏览器优先**：阶段 A/B 必须先调用 `playwright-cli`，不得退回聚合搜索
- **固定技能路径**：`playwright-cli` 技能路径固定为 `/Users/bytedance/.skills-manager/skills/playwright-cli`
- **自动化入口**：优先通过 CLI 执行 `playwright-cli`；如环境支持，也可在 CI 中复用同样的无头浏览器流程

### A2.1 实战补充：为什么之前抓不全，这次为什么抓全了

之前失败的根因，不是官网没数据，而是**只抓了结果页当下渲染出来的 DOM 卡片**。携程国际机票页存在以下特点：

- 列表页顶部会显示类似 `共96个航班，10个直飞`，但当前视口或当前已渲染 DOM 里，可能只看到少量推荐卡片
- `.flight-item.domestic` 这类选择器常常只覆盖当前已渲染子集，不等于整页全部航班
- 登录前、验证码后、或结果页尚未完全拉取时，DOM 更容易只呈现局部内容

这次成功的关键经验是：**先像真人一样登录并进入官网结果页，再从该结果页自己发出的官方搜索接口响应里取全量 itinerary 列表**，而不是只依赖页面上当下可见的卡片。

可复用做法：

1. 先用 `playwright-cli` 打开真实官网结果页，必要时让用户手动完成登录/验证码
2. 登录完成后，保持同一个浏览器 session / profile，不要立刻换新会话
3. 等待结果页继续拉取数据，监听该页面自身触发的官方接口：
   - `international/search/api/search/batchSearch`
   - `international/search/api/search/pull/{searchId}`
4. 以接口返回的 `flightItineraryList` 为“整页全部航班”的准数据源
5. 以页面顶部可见文案（如 `共96个航班，10个直飞`）做交叉核对
6. DOM 只用来补充顶部文案、日期条、当前页面提示；**全量航班表优先取接口返回列表**

重要边界：

- 这种做法仍然满足“官网结果为王”，因为数据来源就是用户眼前这个官网结果页自身发出的请求，不是第三方聚合源
- 如果用户要的是“整页全部航班表”，优先整理 `flightItineraryList`
- 如果用户要的是“每个航班卡片展开后的套餐/舱位细项”，则在 itinerary 基础上继续展开 `priceList`，不要只点几张卡片就停止
- 如果页面顶部总数和接口返回数不一致，必须在文档里注明 `待确认`
- 具体单程 SOP 见：`reference/ctrip-oneway-full-page-sop.md`

### A2.2 往返链路的最小原子操作

往返抓取不要一上来就做全量 `N × N`。先拆成最小原子单位：

1. 打开往返结果页
2. 选择一个去程
3. 等返程阶段加载
4. 从 `routeSearch` 响应中取该去程对应的全部返程组合
5. 先产出 `1 个去程 × N 个返程` 的校验文档

等这个原子链路跑通后，再对所有去程做循环，最终拼成全量 `N × N` 往返组合表。

具体往返 SOP 见：`reference/ctrip-roundtrip-outbound-to-return-sop.md`

如果用户要求“按 loop 逐步写文件、逐步读文件、每轮处理后返回上一页并持续更新 state”，则应切换到：`reference/ctrip-roundtrip-all-outbounds-iterative-sop.md`

补充经验：这条 SOP 已验证可以在同一个 `playwright-cli` session 中连续处理多个去程；推荐做法是 `open` 一次后，每轮只 `goto` 回基础页，不要 `close` / `reopen`。

强约束：
- 只认携程官网实际结果页或详情弹层，不认搜索引擎摘要
- 不允许通过拼接关键词或聚合页估价

### B3. 计算含陆路的总成本

这是阶段 B 的关键——把陆路交通费加进机票价格，算出"门到门"的真实总成本：

```
方案总成本 = 陆路交通（去程+回程）× 人数 + 机票价格 × 人数
```

注意：
- 陆路交通是**往返**的（去的时候要从出发城市到中转城市，回来的时候要从中转城市回出发城市）
- 如果回程航班选择直接飞回出发城市（混合方案），回程就没有陆路交通
